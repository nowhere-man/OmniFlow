## 3. 领域模型

| 文件 | 重点字段 |
|------|----------|
| `Ledger.kt` | id, name, deleted_at, created_at, updated_at |
| `Account.kt` | id, name, type, icon, card_number, note, balance, include_in_total_assets, deleted_at, updated_at |
| `AccountBalanceRecord.kt` | account_id, date, balance, delta, created_at |
| `Transaction.kt` | id, ledger_id, account_id, category_id, amount, type, date, note, is_excluded, external_source, external_id, deleted_at, updated_at |
| `Category.kt` | id, ledger_id, parent_id, name, type, deleted_at, updated_at |
| `Tag.kt` | id, ledger_id, name, deleted_at, updated_at |
| `TransactionTag.kt` | transaction_id, tag_id |
| `Rule.kt` | id, ledger_id, conditions, actions, priority, deleted_at, updated_at |
| `CategoryMemory.kt` | ledger_id, memory_key, category_id, updated_at |
| `RawTransaction.kt` | parser output before rules and dedupe |
| `ImportPreviewItem.kt` | preview row with rule result, dedupe status and user edits |
| `Reminder.kt` | type, amount, schedule fields, paused, deleted_at, updated_at |
| `SyncState.kt` | idle, syncing, success, error |
| `AppError.kt` | code, message, recoverable |
| `ChartData.kt` | chart type, range, series, labels, empty state, drilldown metadata |

默认账户初始化为现金类型、内置图标 key、`include_in_total_assets = true`。账户余额采用混合模式：交易写入更新 `Account.balance`，手动校准写入 `AccountBalanceRecord`。

`RawTransaction` / `ImportPreviewItem` 可保留商户名、交易对方、商品说明用于规则和去重；`Transaction` 不保存这些独立字段，确认入账时统一合并进 `note`。

## 4. Use Case 与 Facade

目录：

```text
domain/usecase/
├── ledger/
├── account/
├── transaction/
├── import/
├── rule/
├── analytics/
├── search/
├── reminder/
├── sync/
└── interop/
```

关键约定：

- 平台优先调用 Facade / Workflow；底层 Use Case 由 Facade / Workflow 编排。
- Use Case 返回 `Result<T>`、`Flow<T>` 或 `Flow<Result<T>>`，错误通过 `AppError` 传递。
- 需要长期状态的流程用 Flow 暴露进度和结果。

### 4.1 导入 Workflow

```kotlin
interface ImportWorkflow {
    fun preview(request: ImportRequest): Flow<Result<ImportPreviewState>>
    suspend fun editItem(edit: ImportPreviewEdit): Result<ImportPreviewState>
    suspend fun commit(sessionId: ImportSessionId): Result<ImportCommitResult>
}
```

内部步骤：

```text
CreateImportPreviewUseCase
  -> ParseBillFileUseCase
  -> ApplyRulesUseCase
  -> ApplyCategoryMemoryUseCase
  -> ApplyDedupeUseCase
  -> SaveImportSessionUseCase

EditPreviewItemUseCase
CommitImportUseCase
```

`ImportSession` 用于预览到入账之间的临时状态。确认入账后清理上传文件、原始解析快照和导入批次记录。

### 4.2 同步 Facade

```kotlin
interface SyncFacade {

    suspend fun syncNow(): Result<SyncResult>

```

## 5. 解析器设计

### 5.1 commonMain Interface

```kotlin
expect class CsvDecoder {
    fun decode(bytes: ByteArray, charset: CsvCharset): String
}

interface BillParser {
    fun parse(bytes: ByteArray): Result<List<RawTransaction>>
}

expect class SpreadsheetParser() {
    fun parseXlsx(bytes: ByteArray): Result<List<RawTransaction>>
    fun parseXls(bytes: ByteArray): Result<List<RawTransaction>>
}
```

CSV 行列解析逻辑放在 commonMain；字符集转换和 spreadsheet 读取使用 expect/actual。

### 5.2 平台实现

| 来源 | commonMain | Android | Apple |
|------|------------|---------|-------|
| 支付宝 CSV/GBK | 行列解析共享 | JVM `Charset.forName("GBK")` | Foundation GB18030 |
| 京东 CSV/UTF-8 BOM | 完全共享 | 同 commonMain | 同 commonMain |
| 美团 CSV/UTF-8 BOM | 完全共享 | 同 commonMain | 同 commonMain |
| 抖音 CSV/UTF-8 BOM | 完全共享 | 同 commonMain | 同 commonMain |
| 青子记账 JSON | 完全共享 | 同 commonMain | 同 commonMain |
| 微信 XLSX | Parser Interface | Apache POI | CoreXLSX |
| 建设银行 XLS/BIFF8 | Parser Interface | Apache POI | 极简 BIFF8 parser |

Apple GBK 解码使用 Foundation 的 GB18030 编码能力，GB18030 是 GBK 超集。

### 5.3 建设银行 XLS 重点实现

基于真实 CCB.xls 样例：

- 全部字段均为纯文本字符串，无数字类型、日期序列号、公式、合并单元格。
- 金额以带千分位字符串存储，去逗号后转金额。
- 日期为 `yyyyMMdd` 文本。
- 结构固定：前 3 行元数据，第 4 行表头，第 5 行起数据。

Apple 极简 BIFF8 parser 只解析：

- SST record `0x00FC`
- LabelSST record `0x00FD`

Apple 首版 BIFF8 parser 的实现范围限定为当前建设银行 fixture 所需记录类型。升级条件：真实样例出现数字、公式、合并单元格或其他未覆盖记录类型。

## 6. 规则、记忆与去重

### 6.1 规则

- Rule conditions 存储为结构化条件列表，规则系统不执行自由文本脚本。
- 首版条件类型：字段包含关键字。
- 首版字段来源：备注、交易类型、来源平台。
- Action 支持：设置一级分类、设置二级分类、设为不计入收支、排除不入账。
- 同账本内按 priority 升序执行，命中后短路。

### 6.2 分类记忆

- 分类记忆以 `ledger_id + memory_key` 为主键。
- `memory_key` 的生成规则首版保持简单：来源平台 + 规范化备注。
- 用户在导入预览页手动选择分类后写入分类记忆。
- 分类记忆只在规则未命中时使用。

### 6.3 去重

- 绝对去重查询 `external_source + external_id`；青子记账导入不作为 `external_source`。
- 模糊去重条件：时间窗口 + 金额相等 + 备注文本相似度超过阈值。
- 普通来源时间窗口为前后 2 小时。
- 建设银行来源窗口为同日。
- 疑似重复默认排除入账。

### 7.2 Adapter Interface

```kotlin
interface SyncAdapter {
    suspend fun listBackups(): Result<List<RemoteBackupMeta>>
    suspend fun uploadBackup(backup: BackupRecord): Result<Unit>
    suspend fun downloadBackup(meta: RemoteBackupMeta): Result<BackupRecord>
    suspend fun deleteBackup(meta: RemoteBackupMeta): Result<Unit>

```

### 7.3 备份记录

```kotlin
data class BackupRecord(
    val deviceId: String,
    val backupId: String,
    val createdAt: Instant,
    val payload: String
)
```

- `payload` 为序列化后的完整业务数据快照，必须包含恢复 App 所需的全部数据。
- 首版备份记录上传前不加密。
- 远端文件名：`backups/{created_at}-{device_id}.backup`。
- `SyncMeta.sq` 记录最近备份时间、远端配置和最大备份记录数。

## 10. 命名与依赖规则

| 类型 | 规范 | 示例 |
|------|------|------|
| Kotlin Entity | PascalCase + 无后缀 | `Transaction.kt` |
| Kotlin Use Case | PascalCase + `UseCase` | `AddTransactionUseCase.kt` |
| Kotlin Facade / Workflow | PascalCase + `Facade` / `Workflow` | `ImportWorkflow.kt` |
| Repository Interface | PascalCase + `Repository` | `TransactionRepository.kt` |
| Repository Implementation | PascalCase + `RepositoryImpl` | `TransactionRepositoryImpl.kt` |
| SQLDelight schema | PascalCase + `.sq` | `Transaction.sq` |
| Swift View | PascalCase + `View` | `TransactionListView.swift` |
| Swift ViewModel | PascalCase + `ViewModel` | `TransactionViewModel.swift` |
| Android Screen | PascalCase + `Screen` | `TransactionListScreen.kt` |
| Android ViewModel | PascalCase + `ViewModel` | `TransactionViewModel.kt` |

依赖规则：

- `domain/` 的依赖范围限定为领域模型和 Repository Interface。
- `data/` 的依赖范围限定为领域模型、Repository Interface、存储和同步实现。
- Platform UI 通过 KMP Facade / Use Case 访问业务能力。
- SQLDelight Query 的调用位置限定在 KMP 数据层。
- 解析器、同步和导入操作必须在后台线程执行。
