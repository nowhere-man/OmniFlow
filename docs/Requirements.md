# OmniFlow Progress
## 当前阶段
正在进行最终产品架构与原生多端路线确认。产品不再以浏览器 Web 或 WebView UI 作为正式交付目标，重点转向 iOS、Android、macOS 三端应用体验与共享业务核心的长期维护方案。
## 已明确需求
### 产品目标
- 开发一个多平台记账 App。
- 最终正式支持 iOS、Android、macOS。
- 不支持浏览器访问、网页部署或 Web 版本。
- Windows 不纳入当前正式目标，除非后续重新确认。
- 目标不是 100% 使用系统原生控件，而是做到精美、成熟、符合各平台主流设计习惯。
- Apple 端采用一个 Apple 工程，iOS 与 macOS 共享 SwiftUI 基础组件、设计 token、数据绑定和图表数据模型，但导航、窗口布局、设置页、列表密度、快捷键等按平台分别设计。
- Android 端首版只支持手机，采用 Material Design / Material 3 设计语言，不单独支持平板或折叠屏。
- Android 与 Apple 允许维护两套 UI 代码，以换取更好的平台体验与更低的长期妥协成本。
- 长期目标是替换当前正在使用的「青子记账」App。
- 近期目标需要支持与「青子记账」数据互通，避免迁移期间被单一 App 绑定。
- 当前讨论重点是明确完整产品的最终形态，不以第一版 MVP 限制最终目标。
- 后续可以分阶段实现，但需求澄清阶段需要先把完整效果讨论清楚。
- 首版只做单用户场景，但必须支持多账本。
- 不做家庭共享账本、多人协作、成员权限、共享邀请或多人冲突合并。
### 基础记账能力
- 支持基本记账功能。
- 交易类型只需要支出和收入。
- 每条交易必须归属于一个账本。
- 手动新增交易必须选择分类；分类从当前账本的分类列表中选择，不使用自由文本填写。
- 支持账户管理；账户是全局共享资源，不从属于任何账本。
- 每条收支需要引用一个全局账户。
- 账户可用于表示支付宝、微信、建设银行储蓄卡、现金、京东支付等资金来源或去向。
- 首次启动 App 时自动创建一个全局「现金」账户；新建账本不会创建账户。
- 导入和手动新增交易可默认使用全局「现金」账户，用户可以在记账页、导入预览页或规则中改为其他全局账户。
- 账户余额不追求精确平账；账户可维护一个用户手动填写的当前余额，交易流水不强制推导账户余额。
- 预算功能首版不做。
### 账本、分类、标签与规则
- 分类按账本独立，不全局共享。
- 标签按账本独立，不全局共享。
- 规则按账本独立，不全局共享。
- 账户全局共享，不按账本独立。
- 新建账本时，从系统预置模板复制一套收入和支出的一级分类，用户后续可在该账本内修改。
- 新建账本时不复制默认规则；规则由用户在对应账本内自行创建。
- 系统预置规则的条件类型和动作类型，用户按需要添加自定义规则。
- 规则条件首版以「字段包含关键字」为主，字段来源包括商户、商品说明、备注、交易对方、交易类型、来源平台等解析出的文本字段。
- 规则动作必须支持设置一级分类、二级分类、标签、全局账户、是否不计入收支、是否排除不入账等导入预览动作。
- 规则不支持把明细记录到其他账本；导入明细始终属于导入前选择的目标账本。
- 规则只在导入预览阶段生效，不对已入账历史交易批量重跑。
- 导入自动分类优先级为：账本内规则 > 同账本历史交易/历史导入选择形成的分类记忆 > 用户手动选择。
- 历史分类记忆只作为规则未命中时的兜底，不替代用户自定义规则。
### 数据分析
- 支持精美、丰富的原生图表展示，用于分析记账数据。
- 支持按天、周、月、年和自定义时间范围查看统计分析。
- 分析口径在共享业务核心中统一维护；不同平台只负责按各自设计语言渲染。
- iOS/macOS 图表遵循 Apple 平台视觉习惯，可优先使用 Swift Charts 与必要的自定义绘制。
- Android 图表遵循 Material 3 视觉习惯，可使用 Compose 图表库或自绘 Canvas。
- 不以 ECharts/WebView 作为正式图表渲染方案。
### 外部账单导入
计划支持以下来源的账单导入：
- 支付宝
- 微信
- 京东
- 美团
- 银行账单，第一阶段暂时只支持建设银行
导入能力需要支持：
- 解析不同平台的账单文件。
- 明细去重。
- 自动或半自动归类。
- 导入前必须先选择目标账本。
- 导入明细默认归属全局「现金」账户；首版不做按账单来源设置默认账户。
- 导入阶段不自动根据支付平台自带分类映射 App 分类。
- 导入预览项如果规则和历史分类记忆都未命中，则分类为空并要求用户手动选择。
- 不允许未分类交易确认入账；所有入账交易必须有分类。
- 用户在导入预览页手动选择分类后，该选择进入同账本历史分类记忆，后续相同或相似明细可自动填分类。
- 确认入账后不保留导入批次记录、原始解析明细快照或上传的原始文件。
- 首版不支持撤销上一次导入。
- 最终交易必须保留 `external_source` 和 `external_id`，用于未来重复导入去重。
- 后续会提供每个支付平台的账单样例文件，格式可能是 CSV 或 JSON 或 XLSX。
### 规则过滤与处理
需要支持账本内用户自定义规则，对导入预览明细进行处理。
- 规则属于账本，创建、编辑、删除和执行都在目标账本上下文内完成。
- 同一导入会话中的所有明细先归属导入前选择的目标账本，规则不得改派账本。
- 规则命中后可以设置分类、账户、标签、不计入收支或排除不入账。
- 规则命中后也可以继续由用户在预览页手动修改，最终以用户确认的预览结果入账。
- 多条规则命中时按用户定义的优先级执行。
- 规则只作用于导入预览，不作用于已入账历史交易。
- 因交易类型只保留支出/收入，转账、退款、还款等导入明细需要通过排除、归类或不计入收支处理，避免污染真实收支。
### 提醒与周期事项
- 提醒与周期事项是独立功能，不绑定账本、账户或交易。
- 提醒与周期事项不自动生成交易，不参与记账统计，不影响账户余额。
- 提醒与周期事项参与多端加密同步。
- 提醒使用本地系统通知，不做服务器推送。
- 提醒分为还款提醒和订阅提醒两个子类型。
- 还款提醒由用户手动维护，金额可选，只用于提醒展示。
- 还款提醒只支持两种规则：固定还款日期、根据账单日后 N 天计算还款日。
- 订阅提醒金额可选，用于展示和未来固定支出概览。
- 订阅提醒支持每天、每周、每月、每年重复。
- 首版不支持复杂 RRULE、法定节假日顺延或工作日规则。
### 青子记账互通
需要支持：
- 导入青子记账导出的 JSON 文件。
- 将本项目数据导出为符合青子记账 JSON 格式的数据，方便再导入青子记账。
待澄清：
- 青子记账 JSON 的完整结构和字段含义。
- 本项目内部数据模型与青子记账模型不完全一致时的映射策略。
- 导出时是否要求 100% 可被青子记账无损导入。
### 技术路线与交付边界
- 最终主线选择「Rust 共享核心 + 平台原生 UI」，详细架构见 `docs/Architecture.md`。
- 后端的正式形态是嵌入在 App 内的 Rust shared core，不优先建设传统云后端或 HTTP 业务后端。
- Rust core 统一维护数据模型、SQLite 存储、账单解析、规则引擎、去重、统计分析、同步和加密能力。
- 本地 SQLite 是每台设备的主数据存储；无网络时记账、导入、搜索和统计分析都应正常工作。
- 远端服务第一阶段只作为加密同步存储使用，首选 WebDAV，不承载明文业务数据、查询、统计或解析逻辑。
- SwiftUI 负责 iOS/macOS UI；Kotlin + Jetpack Compose 负责 Android UI。
- Swift/Kotlin 通过 UniFFI/FFI 或等价桥接方式调用 Rust core。
- Flutter、React Native 可作为未来重新评估的备选，但不作为当前主线，因为用户可以接受两套 UI，且当前目标更重视平台体验和长期维护质量。
- 当前 React/Tauri WebView UI 不作为最终用户端，不继续按 Web 产品优化；目标状态是删除 Web 相关正式交付路径。
- 迁移执行上优先保证 Rust core 与原生壳可承接功能，再移除旧 UI 代码，避免在没有替代入口时中断验证。
## 当前实现假设与约束
- 首版只做单用户多账本。
- 家庭共享账本、多人协作、企业报销、团队财务暂未纳入目标，除非后续明确提出。
- 实现可以分阶段推进，但需求讨论需要覆盖完整产品目标。
- 数据源样例文件会作为解析器设计和测试用例的依据。
- 数据同步方案：本地优先，WebDAV 作为第一优先同步后端。
- 内部同步能力需要抽象为「同步适配器」，未来可扩展 iCloud、坚果云、S3、自建服务等后端。
- 同步到远端的数据默认加密，WebDAV 等远端服务只能看到加密后的数据。
- 本地设备负责解密和使用数据。
未列入「已明确需求」和「技术路线与交付边界」的细节仍需继续确认。

5. 明确数据分析口径：收支趋势、分类占比、账户余额、现金流、异常消费、同比环比等。
6. 明确 Rust core 的 FFI Interface：哪些能力以同步调用暴露，哪些能力需要异步任务、进度回调或事件流。

- (Apple 端决策) iOS 与 macOS 使用一个 Apple 工程，共享 SwiftUI 基础能力、设计 token、数据绑定和图表数据模型，但不强行复用全部界面；iOS/macOS 分别按平台习惯设计导航、布局、密度和交互。
- (Android 端决策) Android 首版只做手机，采用 Material Design / Material 3，不单独设计平板或折叠屏。
- (Web/Tauri 决策) 当前 React/Tauri WebView UI 不再作为最终用户端，不继续按 Web 产品优化。迁移执行上可以先打通 Rust core 与原生壳，再删除 Web/Tauri UI；目标状态是移除 Web 相关正式交付路径。
- (图表目标强化) 明确需要精美、丰富的图表分析。分析口径由共享 Rust core 统一维护，iOS/macOS 和 Android 按各自平台视觉规范渲染，不追求像素级一致，不依赖 ECharts/WebView 作为正式图表方案。
- (后端架构决策) 明确不优先建设传统云后端。Rust shared core 是正式后端，负责本地数据库、业务规则、导入、去重、统计、同步和加密；远端第一阶段只作为加密同步存储，WebDAV 是首选 Adapter。
- (需求细化) 明确首版只做单用户多账本；分类、标签、规则按账本独立；账户全局共享。
- (需求细化) 明确新建账本复制系统预置收入/支出一级分类，不复制默认规则，不创建账户。
- (需求细化) 明确手动新增交易和导入入账都必须有分类；导入预览未分类项不能确认入账。
- (需求细化) 明确导入前必须选择目标账本；规则不支持把明细改派到其他账本。
- (需求细化) 明确导入默认账户为全局「现金」，首版不做按账单来源设置默认账户。
- (需求细化) 明确导入后不保留导入批次、原始解析快照或上传文件，不支持撤销上一次导入；最终交易保留 `external_source` 和 `external_id` 用于去重。
- (需求细化) 明确疑似重复默认不入账，用户手动确认保留才入账；绝对重复锁定不入账。
- (需求细化) 明确提醒与周期事项是独立模块，分为还款提醒和订阅提醒，参与同步，使用本地通知，不自动生成交易。
