## 待澄清队列（迁移自 Requirements.md）

1. 细化本地优先与 WebDAV 同步实现：SQLite schema、同步适配器接口、冲突处理、加密密钥管理和失败重试策略。
2. 明确规则系统的首版字段模板、动作模板、优先级 UI 和规则测试预览方式。
3. 明确历史分类记忆的匹配依据：哪些字段参与匹配，是否区分来源平台，如何处理相似但不完全相同的明细。
4. 明确青子记账互通的兼容等级：临时迁移、长期双写，还是完整双向同步。
5. 明确数据分析口径：收支趋势、分类占比、账户余额、现金流、异常消费、同比环比等。
6. 明确 KMP 层的暴露边界：哪些能力以同步调用暴露，哪些能力需要 suspend/Flow、进度回调或事件流。
7. 明确 iOS/macOS/Android 的首版信息架构与导航方式。
8. 明确原生图表的视觉方向、交互能力和平台差异边界。

## 2026-07-01

- 用户最初提出要开发一个多平台记账 App；平台范围已在 2026-07-08 修正。
- 除基本记账外，需要支持按周、月、年的图表分析。
- 需要支持支付宝、微信、京东、建设银行账单导入。
- 导入后需要支持去重和归类。
- 需要支持用户自定义规则，对导入明细按关键字过滤、排除或执行其他操作。
- 后续会提供各支付平台账单样例，格式可能为 CSV 或 JSON。
- 当前使用「青子记账」，长期希望替换它。
- 项目需要支持导入青子记账 JSON，也支持导出青子记账兼容 JSON。
- 用户明确：现在要讨论完整产品，不一定一次实现，但必须把最终想做成的效果讨论并明确清楚。
- 已修正：首版只做单用户多账本，不做家庭共享账本、多人协作、共享权限或多人冲突合并。
- 已修正：平台目标以 2026-07-08 结论为准，正式支持 iOS、Android、macOS，不再支持 Web/Windows。
- 已确认：同步方案采用本地优先，WebDAV 是第一优先同步后端。
- 已确认：内部仍然抽象成「同步适配器」，未来可以加 iCloud、坚果云、S3、自建服务等。
- 已确认：远端同步数据默认加密，只有本地设备能解密使用。
- 已确认：交易类型只需要支出和收入，不需要借贷、报销等一等交易类型。
- 已确认：需要账户概念，每条收支归属到一个账户。

## 2026-07-02

- (补充讨论) 明确了分类系统（两级）与标签系统（多标签）并存；后续修正为分类、标签、规则均按账本独立。
- (补充讨论) 明确了不单独记录退款；针对转账/报销等场景，需要支持标记一笔明细为「不计入收支」，以便在统计分析中剔除。
- (补充讨论) 明确了与青子记账「长期双写/双向同步」的高互通标准。期望在过渡期内两个 App 都会用，通过导入导出实现双向的数据合并与增量同步。
- (补充讨论) 明确了「流水号绝对去重 + 特征相似度疑似去重 + 手动确认合并」的严谨去重策略，确保多渠道导入和双向同步不产生重复。
- (补充讨论) 明确了规则引擎的边界：丰富的匹配条件和强大的执行动作（改分类、改账户、打标签、排除、标记不计入收支等）；后续修正为规则按账本独立、只作用于导入预览、不支持改派账本。
- (补充讨论) 明确了导入交互流程：采用「一站式预览页」。上传文件自动解析并应用规则后，进入综合列表进行批量确认、修改和去重，最后再一键入账。
- (补充讨论) 明确了数据分析功能：提供全面的收支趋势、分类占比、账户资产分析、同环比分析、标签分析，以及收支排行榜。所有图表均需支持按天、周、月、年和自定义时间范围过滤展示。
- (补充讨论) 明确了预算首版不做。
- (高阶功能) 明确了全局高级搜索与聚合：支持按大类、小类、标签、账户、商户、金额范围、时间段进行任意组合交叉筛选，且搜索结果页能自动计算筛选出来的总收支。
- (高阶功能) 明确了提醒与周期事项：后续修正为独立提醒功能，不自动记账，不绑定账本、账户或交易。
- (高阶功能) 明确了隐私保护（应用锁）：支持启动 App 时要求面容/指纹/密码解锁，保护财务隐私。
- (高阶功能) 明确了还款提醒需求：后续修正为独立提醒子类型，不绑定信用卡/花呗账户。
- (避坑推演结论) 明确了**账户归属**：账户属于全局共享资源，所有账本共用底层的物理账户池（如全局唯一支付宝账户）。
- (避坑推演结论) 明确了**退款处理**：保持极简，不引入复杂的系统平账。未跨月退款无需记录（删原单），跨月退款则手工记一笔"收入"备注退款即可。
- (避坑推演结论) 明确了**余额精度取舍**：用户对各账户（如储蓄卡）的绝对余额精度要求不高。信用卡还款/跨行转账直接不记即可，忽略 App 内账户可能产生的余额偏差，避免引入复杂的"流水关联"机制。
- (避坑推演结论) 已修正**周期账单防重**：首版提醒与周期事项不生成待确认记账卡片，不与导入交易自动合并。

## 2026-07-03

- (账单解析澄清) 确认了最终支持的导入来源为 5 个：**支付宝**(CSV/GBK)、**微信**(XLSX)、**京东**(CSV/UTF-8 BOM)、**美团**(CSV/UTF-8 BOM)、**建设银行**(XLS)。**抖音不再支持**，已从样例中移除。
- (账单解析澄清) 确认了**不需要 PDF 解析**。建设银行已改为导出 XLS 格式，所有 5 个来源均为结构化文件（CSV/XLSX/XLS）。
- (账单解析澄清) 确认了**建设银行的收支判断策略**：金额正负为主（正数=收入，负数=支出），摘要字段（如"消费"、"银联入账"、"转账支取"）为辅佐验证。注意建设银行时间精度仅到天（无时分秒），跨平台模糊去重时需特殊处理。
- (账单解析澄清) 确认了**导入时不解析收/付款方式**。所有导入的明细统一归属到一个「现金」账户，用户后续可通过规则引擎批量修改账户归属。
- (账单解析澄清) 确认了**「不计收支」类交易的处理**：照常导入，自动设置 `is_excluded = true`，用户可在预览页查看并修改。不跳过、不丢弃。
- (账单解析澄清) 已修正**分类来源策略**：不再利用平台自带分类自动映射 App 分类；导入自动分类依次依赖账本内规则、同账本历史分类记忆、用户手动选择。

## 2026-07-08

- (平台目标修正) 明确正式目标改为 iOS、Android、macOS，不再支持浏览器访问、网页部署或 Web 版本；Windows 暂不纳入当前目标。
- (技术路线修正) 明确不追求一套跨端 UI 覆盖所有平台，用户可以接受两套 UI 代码。推荐路线为 KMP 共享领域层 + Apple SwiftUI + Android Kotlin/Jetpack Compose。
- (Apple 端决策) iOS 与 macOS 使用一个 Apple 工程，共享 SwiftUI 基础能力、设计 token、数据绑定和图表数据模型，但不强行复用全部界面；iOS/macOS 分别按平台习惯设计导航、布局、密度和交互。
- (Android 端决策) Android 首版只做手机，采用 Material Design / Material 3，不单独设计平板或折叠屏。
- (Web/Tauri 决策) 当前 React/Tauri WebView UI 不再作为最终用户端，不继续按 Web 产品优化。迁移执行上可以先打通 KMP 层与原生壳，再删除 Web/Tauri UI；目标状态是移除 Web 相关正式交付路径。
- (图表目标强化) 明确需要精美、丰富的图表分析。分析口径由共享 KMP 层统一维护，iOS/macOS 和 Android 按各自平台视觉规范渲染，不追求像素级一致，不依赖 ECharts/WebView 作为正式图表方案。
- (后端架构决策) 明确不优先建设传统云后端。KMP Shared Domain Layer 是正式后端，负责本地数据库、业务规则、导入、去重、统计、同步和加密；远端第一阶段只作为加密同步存储，WebDAV 是首选 Adapter。
- (需求细化) 明确首版只做单用户多账本；分类、标签、规则按账本独立；账户全局共享。
- (需求细化) 明确新建账本复制系统预置收入/支出一级分类，不复制默认规则，不创建账户。
- (需求细化) 明确手动新增交易和导入入账都必须有分类；导入预览未分类项不能确认入账。
- (需求细化) 明确导入前必须选择目标账本；规则不支持把明细改派到其他账本。
- (需求细化) 明确导入默认账户为全局「现金」，首版不做按账单来源设置默认账户。
- (需求细化) 明确导入后不保留导入批次、原始解析快照或上传文件，不支持撤销上一次导入；最终交易保留 `external_source` 和 `external_id` 用于去重。
- (需求细化) 明确疑似重复默认不入账，用户手动确认保留才入账；绝对重复锁定不入账。
- (需求细化) 明确提醒与周期事项是独立模块，分为还款提醒和订阅提醒，参与同步，使用本地通知，不自动生成交易。
