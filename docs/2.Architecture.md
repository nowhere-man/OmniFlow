# OmniFlow Architecture

本文档只描述技术架构和模块职责。产品需求见 `docs/1.Requirements.md`，代码目录与关键接口见 `docs/3.TechDesign.md`，验收标准见 `docs/4.Goal.md`。

---

## 1. 架构结论

OmniFlow 是一个本地优先的纯客户端应用，正式目标平台为 iOS、macOS 和 Android。

核心路线：

- 共享业务核心：Kotlin Multiplatform (KMP)
- 本地数据库：SQLite，由 SQLDelight 生成跨端访问代码
- Apple UI：SwiftUI
- Android UI：Jetpack Compose + Material 3
- 远端同步：支持 iCloud 和 WebDAV；同步采用全量备份记录，超过最大备份记录数时删除最旧记录。
该架构的目的：
- 保持所有业务口径跨端一致：账单解析、规则、去重、统计和同步只实现一次。
- 保持 UI 平台原生：iOS、macOS、Android 可以按各自导航、窗口、手势、快捷键和视觉规范设计。
- 降低服务端成本：远端职责限定为文件存储。

## 2. 总体结构

```text
              User-owned Remote Storage
              (iCloud / WebDAV backups)
                            ▲
                            │ Sync Adapter
             ┌──────────────┴──────────────┐
             │  Shared Domain Layer (KMP)  │
             │                             │
             │ - Domain Models             │
             │ - Repositories              │
             │ - Use Cases / Facades       │
             │ - Parser / Rules / Dedupe   │
             │ - Analytics                 │
             │ - Sync                      │
             │ - SQLDelight Storage        │
             └─────────────┬───────────────┘
                           │
          ┌────────────────┴────────────────┐
          ▼                                 ▼
  Apple Framework                     Android Native
  SwiftUI ViewModels                  AAC ViewModels
  iOS + macOS UI                      Jetpack Compose UI
```

## 3. 职责划分

### Platform UI

平台层负责用户体验和平台集成：
- 导航、窗口布局、菜单栏、快捷键、表单交互、动画、空状态、错误展示。
- 平台原生图表渲染：Apple 使用 Swift Charts 或原生绘制能力，Android 使用 Compose 图表库或 Canvas。
- 系统集成：文件选择、本地通知、生物识别、Keychain/Keystore 接入。
- 平台 ViewModel：Android 使用 AAC ViewModel，Apple 使用 `@Observable` / `ObservableObject`。

平台层通过 KMP 暴露的业务 Interface 访问数据和业务能力；账单解析、规则、去重、统计和同步逻辑集中在 KMP 层。

### Shared Domain Layer

KMP 层是客户端业务核心，负责：

- 领域模型：账本、账户、交易、分类、标签、规则、导入预览、提醒、同步状态等。
- 数据访问：通过 Repository 封装 SQLDelight 查询和写入。
- 业务流程：手动记账、导入预览、确认入账、规则匹配、自动分类记忆、去重、搜索、统计、青子互通。
- 同步：生成完整备份记录，读写远端备份记录，执行备份恢复。
- 对平台暴露 Use Case / Facade / Flow 数据流。

平台 ViewModel 位于平台层。KMP 暴露面向业务流程的 Facade 和 Flow 状态；UI-only 状态、导航事件和生命周期由平台层维护。

---

## 4. 分层规则

KMP 内部分层：

```text
Facade / Use Case
        │
        ▼
Domain Model + Repository Interface
        ▲
        │
Repository Implementation + Local Storage + Sync Adapter
```

依赖规则：

- `domain` 的依赖范围限定为领域模型和 Repository Interface。
- `data` 的依赖范围限定为领域模型、Repository Interface、存储和同步实现。
- 平台 UI 只调用 KMP 暴露的 Use Case、Facade、Repository Interface 或 Flow。
- SQLDelight 查询只在 KMP 数据层内使用。

---

## 5. 数据架构

- 每台设备的 SQLite 是主数据源。
- 所有实体使用 UUID 字符串主键，禁止依赖数据库自增 ID。
- 账本、账户、交易、分类、规则等支持软删除；软删除记录参与同步。
- 交易必须携带 `ledger_id`，账本是交易列表、搜索、统计的默认过滤边界。
- 账户是全局资源，不挂在账本下。
- 默认账户的类型为现金，默认计入总资产；图标使用内置图标 key。
- 账户余额采用混合模式：交易自动更新余额，用户可手动校准当前余额。
- 分类、标签、规则、历史分类记忆按账本隔离。
- 标签使用交易-标签关联表，支持筛选和聚合。
- Schema 和迁移由 SQLDelight `.sq` 文件维护。

## 6. 同步架构

同步采用本地优先、远端完整备份模型：

- 所有写入先落本地 SQLite。
- 每台设备首次启动生成稳定 `device_id`。
- 同步上传单元为全量备份记录，包含恢复 App 所需的全部业务数据。
- 远端仅按文件存储备份记录，不执行查询、解析或冲突判断。
- 用户可配置最大备份记录数；超过上限时删除最旧备份记录。
- 恢复同步时，本地设备下载用户选择的备份记录并恢复本地数据库。
- 删除 App 后重新安装，连接同一同步目标并恢复备份记录后，数据应与上传前一致。
- 首版不做跨设备自动合并；多设备产生多个备份记录，恢复以用户选择的备份记录为准。

## 8. 导入架构

导入流程在 KMP 内部统一实现：

```text
File bytes
  -> Parser
  -> RawTransaction
  -> Rules
  -> Category Memory
  -> Dedupe
  -> Import Preview
  -> Commit
  -> Transaction
```

平台层只负责文件选择、进度展示、用户编辑和错误映射。平台层不拼装解析、规则、去重流程。

导入解析出的商户名、交易对方、商品说明只保留在导入预览态；确认入账时合并写入交易备注，不作为独立交易字段。

解析策略：

- CSV 行列解析尽量放在 commonMain 共享，字符集转换使用 expect/actual。
- XLSX/XLS 使用 expect/actual 分平台实现。
- 所有解析在后台线程执行，主线程只接收进度和结果。

账单解析的具体库、接口和平台实现放在 `docs/3.TechDesign.md`。

---

## 9. 图表与搜索架构

- 统计口径由 KMP 统一维护。
- KMP 通过 SQL 聚合或领域逻辑输出稳定的 ChartData / Summary 数据模型。
- 平台 UI 不从交易明细二次计算统计口径，只负责渲染和交互。
- 搜索在 KMP 层执行组合查询，并返回交易列表与收入/支出聚合结果。

---

## 10. Apple 接入

- KMP 通过 Kotlin/Native 编译为 Apple Framework。
- Apple 使用 SKIE 将 Kotlin `suspend fun` 和 `Flow` 转换为 Swift 习惯用法。
- Apple ViewModel 保留在 Swift 侧，调用 KMP Facade / Use Case。
- 已知成本：Kotlin/Native 冷编译慢、Framework 含 K/N 运行时、Xcode 调试 KMP 层体验受限。
- 对策：KMP 业务逻辑和解析器优先用 JVM 单元测试覆盖。

---

## 11. 青子互通架构

青子记账互通是独立的导入/导出转换模块，与同步模块分离。

- 同步解决 OmniFlow 多设备一致性。
- 青子互通解决与外部 App 的迁移和长期双写。
- 青子 JSON 导入后进入标准导入流水线，但不作为交易来源枚举值。
- 青子 JSON 导出从本地领域模型转换为青子兼容格式。
- 未映射字段记录警告并跳过，导入或导出流程继续执行。
