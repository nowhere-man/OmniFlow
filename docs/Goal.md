# OmniFlow 验收目标
本文档定义 OmniFlow 的全量验收标准。每一条目标都必须可验证（通过/不通过），不允许模糊措辞。
---
## 零、平台与架构验收目标
### 0.1 平台范围
- [ ] 正式目标平台仅包含 iOS、Android、macOS。
- [ ] 不提供浏览器访问、网页部署、PWA 或 Web 版本验收入口。
- [ ] Windows 不纳入当前正式验收目标，除非后续重新确认。
- [ ] iOS 首版以 iPhone 体验验收，iPad 不单独设计或验收。
- [ ] Android 首版以手机体验验收，平板和折叠屏不单独设计或验收。
- [ ] macOS 作为正式桌面端验收，必须使用桌面导航、窗口、菜单、快捷键和信息密度，不允许直接拉伸手机界面。

### 0.2 技术路线
- [ ] 项目主线采用「Rust 共享核心 + 平台原生 UI」。
- [ ] Rust shared core 作为正式后端实现，不要求首版建设传统云后端、HTTP 业务后端、Postgres 或 MySQL。
- [ ] Rust core 统一维护数据模型、SQLite 存储、账单解析、规则引擎、去重、统计分析、同步和加密能力。
- [ ] 本地 SQLite 是每台设备的主数据存储；无网络时基础记账、导入预览、搜索和统计分析均可使用。
- [ ] Apple 端使用 SwiftUI 开发 iOS/macOS UI，可共享基础组件、设计 token、数据绑定和图表数据模型。
- [ ] Android 端使用 Kotlin + Jetpack Compose + Material 3 开发 UI。
- [ ] Swift/Kotlin 通过 UniFFI/FFI 或等价桥接方式调用 Rust core。
- [ ] 平台 UI 不直接访问数据库，不重复实现账单解析、去重、规则、统计、同步和加密逻辑。
- [ ] FFI Interface 面向用例暴露能力，例如导入预览、确认入账、查询交易、生成图表数据和执行同步，不暴露原始表级 CRUD 作为主要调用方式。
- [ ] Flutter、React Native 不作为当前主线技术栈；只有在未来重新评估并修改本文档后才可引入。
- [ ] React/Tauri WebView UI 不作为最终用户端，正式交付包不包含 Web/Tauri UI 入口。

### 0.3 平台体验与视觉质量
- [ ] iOS UI 遵循 Apple HIG 的导航、表单、弹层、手势、动态字体、安全区和系统反馈习惯。
- [ ] macOS UI 遵循 Apple 桌面应用习惯，支持侧边栏/多栏布局、菜单栏、键盘快捷键、窗口尺寸变化和鼠标/触控板交互。
- [ ] Android UI 遵循 Material 3 的导航、组件、状态层、动效、排版、图标和触控反馈习惯。
- [ ] 各平台允许视觉和交互存在平台差异，不要求像素级一致；但业务信息架构、统计口径和数据结果必须一致。
- [ ] 首页、交易列表、导入预览、数据分析、设置页在各平台验收时必须具备完整空状态、加载态、错误态和长内容适配。

## 一、需求验收目标
### 1. 基础记账
- [ ] 支持手动新增一条收入或支出记录，包含：金额、分类、账户、备注、商户、标签、日期。
- [ ] 手动新增交易必须先处于一个目标账本上下文内，并从该账本的分类列表中选择分类。
- [ ] 手动新增交易不允许保存为未分类。
- [ ] 手动新增交易允许 `external_source` 和 `external_id` 为空。
- [ ] 支持编辑已有交易的任意字段。
- [ ] 支持删除交易（软删除，标记 `deleted_at`）。
- [ ] 支持将一笔交易标记为「不计入收支」(`is_excluded = true`)，标记后该笔交易在统计图表中不参与计算。
- [ ] 交易类型仅支持 `收入` 和 `支出`，不存在第三种类型。

### 2. 多账本
- [ ] 首版只支持单用户多账本，不支持家庭共享账本、多人协作、成员权限或共享邀请。
- [ ] 支持创建、编辑、删除（软删除）多个账本。
- [ ] 每条交易必须归属于一个账本（`ledger_id` 非空）。
- [ ] 所有查询接口（交易列表、统计、搜索）均以账本为边界过滤。
- [ ] 首次启动时自动创建一个「默认账本」。
- [ ] 新建账本时自动复制系统预置的收入/支出一级分类模板。
- [ ] 新建账本时不复制默认规则。
- [ ] 新建账本时不创建账户。

### 3. 账户管理
- [ ] 账户为全局共享资源，不从属于任何账本。
- [ ] 支持创建、编辑、删除（软删除）账户。
- [ ] 账户类型支持：现金、储蓄卡、信用卡、电子钱包、其他。
- [ ] 首次启动时自动创建一个「现金」账户。
- [ ] 账户可维护一个用户手动填写的当前余额。
- [ ] 账户余额不要求从交易流水自动推导，也不要求与交易流水精确平账。
- [ ] 还款提醒不绑定账户，账户模型不包含还款提醒字段。

### 4. 分类与标签
- [ ] 分类按账本独立，不全局共享。
- [ ] 分类为两级树结构：一级分类（父）和二级分类（子）。
- [ ] 系统内置以下常用的一级分类模板，新建账本时复制到该账本，用户可在账本内修改：
  - **支出类**：餐饮美食、交通出行、日用百货、居住服务、休闲娱乐、医疗健康、教育培训、充值缴费、其他支出。
  - **收入类**：工资、投资理财、兼职副业、奖金红包、其他收入。
- [ ] 每个分类有固定的类型属性（`expense` 或 `income`），不可混用。
- [ ] 每条已入账交易必须关联一个当前账本内的分类，不允许保存为未分类。
- [ ] 标签按账本独立，不全局共享。
- [ ] 标签为扁平多标签系统，存储为 JSON 数组。
- [ ] 一条交易可以打多个标签，也可以不打标签。


### 5. 账单导入与解析

#### 5.1 支付宝解析器
- [ ] 能正确解析 GBK 编码的 CSV 文件。
- [ ] 自动跳过文件头部的元数据行（导出信息、特别提示等），定位到数据表头行。
- [ ] 正确提取字段：交易时间、交易分类、交易对方、商品说明、收/支、金额、交易状态、交易订单号。
- [ ] 用 `examples/支付宝.csv` 作为测试 fixture，全部 107 条记录解析成功且字段映射正确。
- [ ] 「不计收支」类交易自动设置 `is_excluded = true`。
- [ ] 支付宝自带的「交易分类」字段只作为来源字段保留，不自动映射为 App 分类。
- [ ] 所有导入明细统一归属到「现金」账户。

#### 5.2 微信解析器
- [ ] 能正确解析 XLSX 格式文件。
- [ ] 自动跳过文件头部的元数据行，定位到数据表头行（交易时间、交易类型、交易对方...）。
- [ ] 正确提取字段：交易时间、交易类型、交易对方、商品、收/支、金额、当前状态、交易单号。
- [ ] 用 `examples/微信.xlsx` 作为测试 fixture，全部 86 条记录解析成功。
- [ ] 「中性交易」类交易自动设置 `is_excluded = true`。
- [ ] 微信无自带分类字段时，分类由规则或历史分类记忆填充；未填充前不得确认入账。

#### 5.3 京东解析器
- [ ] 能正确解析 UTF-8 BOM 编码的 CSV 文件。
- [ ] 自动跳过文件头部的元数据行，定位到数据表头行。
- [ ] 正确提取字段：交易时间、商户名称、交易说明、金额、交易状态、收/支、交易分类、交易订单号。
- [ ] 用 `examples/京东.csv` 作为测试 fixture，全部 16 条记录解析成功。
- [ ] 「不计收支」类交易自动设置 `is_excluded = true`。
- [ ] 京东自带的「交易分类」字段只作为来源字段保留，不自动映射为 App 分类。

#### 5.4 美团解析器
- [ ] 能正确解析 UTF-8 BOM 编码的 CSV 文件。
- [ ] 自动跳过文件头部的元数据行，定位到数据表头行（交易创建时间、交易成功时间...）。
- [ ] 正确提取字段：交易成功时间、交易类型、订单标题、收/支、订单金额、实付金额、交易单号。
- [ ] 用 `examples/美团.csv` 作为测试 fixture，全部 18 条记录解析成功。
- [ ] 美团无自带分类字段时，分类由规则或历史分类记忆填充；未填充前不得确认入账。

#### 5.5 建设银行解析器
- [ ] 能正确解析 XLS（Excel 97-2003）格式文件。
- [ ] 自动跳过前 3 行元数据（标题、账号信息、合计），从第 4 行表头开始解析。
- [ ] 正确提取字段：摘要、交易日期、交易金额、账户余额、交易地点/附言、对方账号与户名。
- [ ] 收支判断：金额正数 = 收入，金额负数 = 支出。
- [ ] 用 `examples/建设银行.xls` 作为测试 fixture，全部 152 条记录解析成功。
- [ ] 建设银行无自带分类字段时，分类由规则或历史分类记忆填充；未填充前不得确认入账。
- [ ] 交易时间仅精确到天（`20260601` 格式），解析为当日 `00:00:00`。

#### 5.6 通用解析要求
- [ ] 每个解析器输出统一的 `Vec<RawTransaction>` 结构，下游规则引擎和去重引擎无需感知来源差异。
- [ ] 文件格式不匹配时返回明确错误（如把微信 XLSX 当支付宝 CSV 解析）。
- [ ] 交易订单号正确提取为 `external_id`，用于后续去重。
- [ ] 导入前必须选择目标账本，导入会话中的明细不得被规则改派到其他账本。
- [ ] 所有导入明细默认归属全局「现金」账户。
- [ ] 首版不支持按账单来源设置默认账户。
- [ ] 确认入账后的最终交易必须保留 `external_source` 和 `external_id`。
- [ ] 确认入账后不保留导入批次记录、原始解析明细快照或上传的原始文件。
- [ ] 首版不支持撤销上一次导入。

### 6. 规则引擎
- [ ] 规则按账本独立，不全局共享。
- [ ] 规则只在导入预览阶段执行，不对已入账历史交易批量重跑。
- [ ] 支持的匹配条件：解析出的文本字段包含关键字，例如商户名、商品说明、备注、交易对方、交易类型、来源平台。
- [ ] 支持的执行动作：设置一级分类、设置二级分类、改账户、加标签、设为不计入收支、排除不入账。
- [ ] 规则不支持把明细记录到其他账本。
- [ ] 规则按用户定义的优先级顺序执行，命中后短路（不再匹配后续规则）。
- [ ] 规则的 CRUD（创建、读取、更新、删除）。
- [ ] 用户可在导入预览页覆盖规则结果，最终以用户确认的预览结果入账。
- [ ] 规则未命中时，系统使用同账本历史交易/历史导入选择形成的分类记忆自动填分类。
- [ ] 历史分类记忆仍未命中时，预览项分类为空，用户必须手动选择分类后才能入账。
- [ ] 用户在预览页手动选择分类后，该选择进入同账本历史分类记忆。

### 7. 去重引擎
- [ ] **绝对去重**：相同 `external_id` + `external_source` 的交易直接标记为重复，不入账。
- [ ] **模糊去重**：时间窗口(±2 小时) + 金额相等 + 商户名相似度 > 阈值 → 标记为「疑似重复」。
- [ ] 建设银行来源因时间仅精确到天，模糊去重窗口放宽为同日。
- [ ] 疑似重复项默认不入账，用户手动确认保留后才允许入账。
- [ ] 去重引擎作为独立模块，与规则引擎解耦。

### 8. 导入预览流程（一站式预览页）
- [ ] 用户上传文件后，系统自动完成：解析 → 规则匹配 → 去重检测。
- [ ] 预览页展示完整的待入账明细列表，每条明细显示：日期、金额、商户、分类、标签、是否排除、来源平台。
- [ ] 重复项（绝对重复和疑似重复）在预览页有醒目标识。
- [ ] 用户可在预览页逐条编辑（修改分类、账户、标签、备注、是否排除）。
- [ ] 用户可在预览页批量操作（全选/反选、批量改分类、批量排除）。
- [ ] 存在分类为空且未排除的预览项时，不允许一键入账。
- [ ] 用户确认后一键入账，所有通过确认的明细写入数据库。
- [ ] 被标记为绝对重复的明细默认不入账，用户无法强行入账。

### 9. 数据分析与图表
- [ ] **收支趋势图**：折线图展示指定时间范围内的收入/支出趋势。
- [ ] **分类占比图**：饼图/环形图展示各分类的支出或收入占比。
- [ ] **账户资产分析**：展示各账户的资产分布。
- [ ] **同环比分析**：当月 vs 上月、当月 vs 去年同月的收支对比。
- [ ] **标签分析**：按标签维度汇总收支。
- [ ] **收支排行榜**：按金额排序的 Top N 交易列表。
- [ ] 所有图表支持按天、周、月、年、自定义时间范围进行过滤。
- [ ] 所有聚合计算在 Rust core 中通过 SQL 或领域逻辑完成，平台 UI 不从交易明细二次计算统计口径。
- [ ] Rust core 通过 FFI 返回稳定的图表数据模型，包含时间粒度、单位、序列、分类、空状态和可下钻标识。
- [ ] iOS/macOS 图表使用 Swift Charts 或 Apple 原生绘制能力渲染，不使用 WebView 图表作为正式方案。
- [ ] Android 图表使用 Compose 图表库或 Canvas 渲染，不使用 WebView 图表作为正式方案。
- [ ] 图表支持触点/鼠标选择、图例切换、时间范围切换、分类下钻和筛选联动。
- [ ] 图表在空数据、少量数据、极值数据和密集数据下均保持标签可读、布局不重叠、动画不过度干扰阅读。
- [ ] 统计结果自动排除 `is_excluded = true` 的交易。

### 10. 高级搜索与聚合
- [ ] 支持按以下维度任意组合筛选：一级分类、二级分类、标签、账户、商户关键字、金额范围、时间范围。
- [ ] 搜索结果页展示匹配的交易列表。
- [ ] 搜索结果页自动计算并展示筛选范围内的总收入和总支出。
- [ ] 搜索在 Rust core 中执行 SQL 查询，不在平台 UI 中过滤。

### 11. 提醒与周期事项
- [ ] 提醒与周期事项是独立功能，不绑定账本、账户或交易。
- [ ] 提醒与周期事项不自动生成交易，不参与记账统计，不影响账户余额。
- [ ] 提醒与周期事项支持创建、编辑、暂停、删除。
- [ ] 提醒与周期事项参与多端加密同步。
- [ ] 提醒使用本地系统通知，不使用服务器推送。
- [ ] 提醒分为还款提醒和订阅提醒两个子类型。
- [ ] 还款提醒金额可选，只用于提醒展示。
- [ ] 还款提醒支持固定还款日期。
- [ ] 还款提醒支持根据账单日后 N 天计算还款日。
- [ ] 订阅提醒金额可选，用于展示和未来固定支出概览。
- [ ] 订阅提醒支持每天、每周、每月、每年重复。
- [ ] 首版不支持复杂 RRULE、法定节假日顺延或工作日规则。

### 12. 青子记账互通
- [ ] 准确解析青子记账 JSON 格式的核心实体：账本 (`bookJsonString`)、账户 (`accountJsonString`)、分类 (`categoryJsonString`)、标签 (`markJsonString`)、交易 (`entryJsonString`)。
- [ ] **数据导入映射规则**：
  - `entry.value` 映射为金额。
  - `entry.createDate` (Unix时间戳) 映射为交易时间。
  - `entry.content` 映射为备注。
  - 根据 `category.type` (0=收入, 1=支出) 确定交易类型。
  - `entry.categoryID` 关联分类名称；若为平级分类直接映射为一级分类。
  - `entry.accountID` 关联账户名称。
  - `entry.markIDs` (逗号分隔的标签ID) 关联并展开为标签文本数组。
  - `entry.excludeFromBudget = true` 时，自动映射为 `is_excluded = true`。
- [ ] 支持将本项目数据导出为青子记账兼容的 JSON 格式。
- [ ] 无法映射的自定义字段（如位置信息、退款周期）在导入时直接丢弃，不阻断流程。
- [ ] 导入导出支持基于时间范围或重复检测的增量操作，满足长期双写的需求。
- [ ] 支持导入青子记账 JSON 文件。导入后进入标准导入流水线（规则 → 去重 → 预览 → 入账）。
- [ ] 支持将本项目数据导出为青子记账 JSON 格式。
- [ ] 无法映射的字段在导入时记录警告但不阻断，导出时同理。
- [ ] 导入导出支持增量操作（基于交易 ID 或时间范围），满足长期双写的需求。
- [ ] 用 `examples/青子记账.json` 作为测试 fixture，记录全部解析成功。

### 13. 数据同步（WebDAV）
- [ ] 本地优先：无网络时所有功能正常工作，有网络时自动/手动同步。
- [ ] 远端第一阶段只作为加密同步存储，不承载明文业务数据、查询、统计、导入解析或规则执行。
- [ ] Rust core 负责 WebDAV 同步、同步适配器抽象、远端加密和本地解密。
- [ ] 同步 Module 通过 Adapter Interface 接入 WebDAV，未来新增 iCloud、S3、自建存储时不改变业务 Module 的 Interface。
- [ ] 上传到 WebDAV 的数据默认为加密 payload，WebDAV 服务端无法读取账本、交易、分类、标签和统计内容。
- [ ] FFI 边界统一返回类型化 `Result<T, AppError>` 或等价错误模型，错误包含稳定错误码、用户可读消息和可恢复性标记。
- [ ] Swift/Kotlin 根据错误类型映射为平台原生 UI 状态：可恢复错误用轻量提示，业务错误用弹层/表单内提示，不可恢复错误用全屏错误页。
